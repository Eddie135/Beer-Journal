# BEER JOURNAL 长期开发规则

本文件适用于仓库中的所有目录和所有后续开发任务。除非用户明确改变需求，否则 Codex 和其他协作者必须长期遵守这些规则。

## 1. 用户与沟通

- 用户没有编程经验。汇报使用简明中文，先说明结果，再说明验证方式和下一步。
- 不要求用户手工编辑代码、配置或数据库。能由 Codex 完成的修改必须由 Codex 完成。
- 必须让用户知道：本次改了什么、没有改什么、实际运行了哪些检查、是否有未解决问题。
- 需要用户操作时，明确写出在哪个程序、哪个目录执行，以及正常结果是什么；不要只给出专业术语。
- 不能把“理论上能运行”写成“已经验证”。没有实际运行的检查必须明确标注为“未运行”及原因。

## 2. 开始修改前

1. 阅读 `README.md`、本文件和与任务有关的 `docs/` 文档。
2. 检查当前目录、Git 状态、相关代码、测试和最近提交。
3. 保留用户已有的未提交改动，不覆盖、不回退、不清理与当前任务无关的内容。
4. 用一句话说明本次目标、预期影响范围和明确不做的内容。
5. 如果需求会改变数据含义、删除策略、安全边界或整体架构，先分析影响并更新相应设计文档，再实施代码。

## 3. 范围与架构边界

- 第一版采用 Django 单体架构：Django 服务端模板、本地静态样式、少量原生 JavaScript、PostgreSQL、Pillow、Gunicorn、Caddy 和 Docker Compose。
- 第一版不建立独立 Vue/React 前端，不建立公开 REST API；确实需要原生 App 时再评估 API。
- 未经用户明确同意并更新 `docs/ARCHITECTURE.md`，不得改换框架或加入 Redis、消息队列、微服务、Kubernetes、Elasticsearch等额外服务。
- 第一版 PWA 只做可安装、应用式显示、静态资源缓存和离线提示；不得自行扩展为离线新增、离线编辑或同步冲突处理。
- 每次只推进一个明确阶段或一个紧密相关的功能集合。不得用“顺便优化”为理由大规模重构。
- 生产环境必须使用 PostgreSQL；测试数据库也应使用 PostgreSQL，不能只在 SQLite 上验证后声称可部署。

## 4. 数据规则

- `Beer` 表示一款啤酒的相对稳定资料；`Tasting` 表示一次实际品饮。日期、地点、价格、容量、照片、笔记和评分属于 `Tasting`，不得塞回 `Beer`。
- 同一款 `Beer` 必须允许关联多条 `Tasting`，修改一次品饮不得覆盖其他历史品饮。
- 动态评分必须通过“评分维度 + 品饮评分值”关系保存，不得把香气、味道等写成七个永久固定字段。
- 历史评分必须保存评分维度名称和量表快照；停用或调整维度不得悄悄改变旧记录的含义。
- 金额和评分使用十进制类型，不能使用浮点数；未知值使用 `NULL`，不能用 0 代替未知。
- 数据库结构变化必须通过迁移文件完成。不得直接手工改生产数据库结构。
- 永久删除有历史记录的数据必须有明确影响说明和二次确认。常规删除优先采用可恢复的逻辑删除。
- 照片路径必须是相对存储键，不能保存 Windows 或 Debian 的绝对路径。

## 5. 安全与隐私

- 真实密码、服务器地址中的秘密部分、数据库口令、API Key、Cookie 和私钥不得进入代码、示例、日志、测试数据、文档或 Git 历史。
- 秘密仅通过未提交的环境变量或服务器秘密文件提供；示例配置只使用显眼的占位值。
- 正式部署必须启用登录和 HTTPS，关闭公开注册；所有私人页面、导出文件和照片都必须经过登录校验。
- 不把用户的啤酒记录、照片或使用统计发送给第三方服务，除非用户针对具体服务明确授权。
- 上传图片必须限制文件大小和像素数、实际解码验证、重新编码并使用随机文件名；不能只相信扩展名或浏览器提供的 MIME 类型。
- 不在日志中记录密码、会话、完整请求体或照片内容。
- 部署前必须运行 Django 的生产安全检查，并解决高风险警告或明确记录例外理由。

## 6. 照片处理

- 第一版照片归属于某次 `Tasting`；`Beer` 页面通过关联的品饮汇总照片。
- 默认只保留重新编码后的展示图和缩略图，不永久保留手机原图，以控制空间并移除 EXIF 隐私数据。
- 写入流程必须避免半成品：解码、旋转、压缩成功后才保存正式数据库记录；失败时清理临时文件。
- 删除数据库照片记录后必须清理对应文件；同时提供孤儿文件检查，避免长期积累。
- 用户上传的照片不能由未鉴权的公开 `/media` 地址直接访问，也不能被 Service Worker 长期缓存。

## 7. 测试与验证

每个功能至少覆盖其风险对应的检查：

- 模型关系、数据库约束和迁移；
- 表单校验、权限、成功路径和失败路径；
- 筛选、排序、统计等查询结果；
- 图片压缩、错误文件、文件清理；
- 导出格式和备份恢复；
- 手机与桌面的关键浏览器流程。

每个稳定阶段完成前必须：

1. 运行 Django 系统检查。
2. 运行全部自动测试，不能只运行刚新增的测试。
3. 检查是否存在未生成或冲突的迁移。
4. 实际启动应用并完成该阶段的人工冒烟流程；能自动化的关键流程应逐步加入浏览器测试。
5. 更新 README 当前状态和相关设计文档。
6. 汇报执行过的命令类别、通过数量、失败数量及未执行项。

测试失败时先保留完整错误信息并分析根因。不得通过删除测试、放宽正确的断言、吞掉异常或关闭安全检查来制造“通过”。

## 8. Git 规则

- 开始和结束修改时都检查 Git 状态及差异。
- 不使用破坏性回退命令，不擅自删除或覆盖用户的未提交更改。
- 每个提交只包含一个可说明的稳定阶段；文档、迁移和对应测试应与功能一起提交。
- 自动测试失败、迁移不一致或存在已知数据损坏风险时不得创建“稳定”提交。
- 提交信息使用简洁明确的格式，例如 `docs: 完成初始产品与架构设计`、`feat: 增加品饮记录`。
- 任何秘密意外进入 Git 后，不能只删除当前文件；必须立即停止并向用户说明需要轮换秘密和清理历史。

## 9. 文档同步

- 产品范围或验收标准变化时更新 `docs/PRODUCT_REQUIREMENTS.md`。
- 技术栈、服务边界、安全或部署拓扑变化时更新 `docs/ARCHITECTURE.md`。
- 表、字段、关系、约束、索引、删除或导出结构变化时更新 `docs/DATABASE_DESIGN.md`。
- 阶段状态、顺序或完成条件变化时更新 `docs/DEVELOPMENT_PLAN.md`。
- `README.md` 只保留项目介绍、当前状态和下一步，不堆积详细开发日志。

## 10. 完成定义

只有同时满足以下条件，才能向用户报告一个阶段“完成”：

- 验收标准全部满足；
- 代码、数据库迁移、测试和文档一致；
- 自动测试及实际操作验证均通过；
- 没有未说明的安全或数据丢失风险；
- Git 差异已检查并创建稳定提交；
- 用户收到可理解的结果、验证方法、已知限制和下一步说明。
