# BEER JOURNAL 分阶段开发计划

文档状态：v2.0 重构执行计划，待实现
更新日期：2026-07-12
目标：完成手机优先的个人啤酒收藏 App v2.0，再进入部署与发布阶段

## 1. 执行原则

每个阶段都必须遵循同一流程：

1. 开始前说明本阶段目标、预计影响文件和明确不做内容。
2. 先检查现有代码、文档、测试和 Git 状态，不覆盖用户改动。
3. 一次只处理当前阶段，不跨阶段同时堆叠功能。
4. 完成数据库迁移检查、自动测试和一轮实际浏览器冒烟验证。
5. 向用户提供非专业语言的可见验收步骤及预期结果。
6. 更新 README 和受影响的设计文档。
7. 检查 Git 差异，建立一个可回退的稳定提交。
8. 只有当前阶段通过，才进入下一阶段；失败则先修复或回退。

## 2.1 v2.0 重构边界

v2.0 是在既有 Beer、Tasting、Photo、标签和评分历史之上演进，不通过删除表或清空历史来简化 UI。实施顺序必须先做数据迁移与兼容测试，再做界面重构。

- UI 移除首页、主导航回收站、食物搭配、饮用场景和多维评分录入；数据层继续保存并可导出历史值。
- BeerStyle 升级为 BeerCategory → BeerStyle 两级分类；迁移须为已有 Style 指定明确类别，不能产生无分类 Beer。
- 新增 Beer 的 Plato、口感档案，以及 Tasting 的容量、饮用瓶数、购买渠道字段；已有 `volume_ml` 已无损重命名为 `capacity`，`purchase_channel` 已规范为新的选项并处理历史值，禁止重复存储同一含义。
- 风味标签支持用户录入自定义值，依靠规范化唯一约束去重；迁移前需检查现有重复标签并报告处理方式。
- v2.0 不开发高级搜索、AI、条码、库存或复杂权限系统。

- 历史评分维度快照保护。

### 自动验证

- 允许不评分或只填部分维度；
- 范围和 0.5 步进校验；
- 一次品饮同一维度不能重复；
- 停用或改名测试维度后，旧评分仍显示原快照；
- 标签规范化后不会重复。

### 用户可见验收

为两次品饮填写不同评分；停用一个测试维度后，新表单不再显示它，但旧记录仍能正确展示。

### 本阶段不做

- 自动加权总分；
- 多套评分模板；
- 品饮级场景标签。

## 7. 阶段 4：多图上传与压缩

### 目标

在手机上安全上传多张照片，并控制磁盘占用。

### 交付物

- Photo 模型和迁移；
- 多选、预览、上传、排序和删除；
- JPEG/PNG/WebP 解码、方向修正、EXIF 移除、尺寸限制和重新编码；
- 展示图与缩略图；
- 登录保护的照片读取；
- 文件清理和孤儿检查工具。

### 自动验证

- 正常横图、竖图、透明 PNG 和 WebP；
- 损坏文件、伪装扩展名、超大体积和超大像素被拒绝；
- 压缩失败不留下 Photo 或临时文件；
- 未登录不能读取照片；
- 永久删除后实际文件被清理。

### 用户可见验收

使用真实手机上传多张照片，页面正确显示方向和缩略图；服务器保存的展示图明显小于原图，且移除位置等 EXIF 信息。

### 本阶段不做

- HEIC/HEIF 承诺；
- 云存储或对象存储；
- 永久保留手机原图。

## 8. 阶段 5：搜索、组合筛选、排序与分页

### 目标

可靠地从品饮历史中找到目标记录。

### 交付物

- 啤酒库的基础搜索；
- 品饮日志的名称、品牌、酒厂、国家、类型、标签搜索；
- 评分、酒精度、价格和日期区间组合筛选；
- 排序、稳定次排序、分页、结果数和清空条件；
- 可复现的筛选样本数据。

### 自动验证

- 每个单项条件及代表性组合；
- 不同类别 AND、同类多选 OR；
- 区间边界、空值、CNY 价格语义；
- 排序空值置后及翻页稳定性；
- 无结果和非法参数不会报错。

### 用户可见验收

使用固定样本完成“德国或比利时、IPA、总分不低于 8、价格不高于 50 元、指定日期范围”的组合筛选，并核对结果。

### 本阶段不做

- 全文搜索服务；
- 自然语言查询；
- 动态维度区间筛选，除非总体评分筛选已稳定且时间允许。

## 9. 阶段 6：统计、导出、备份与恢复

### 目标

让数据可理解、可带走、可从故障中恢复。

### 交付物

- 啤酒款数、品饮次数、国家数、常喝类型、平均分和月度数量；
- 版本化 JSON 导出；
- 分表 CSV 导出包；
- PostgreSQL + 照片 + 清单 + 校验值的完整备份；
- 恢复到空环境的自动化步骤和记录。

### 自动验证

- 用固定数据人工可算地核对统计；
- JSON 可解析、关系和 UUID 完整；
- CSV 表头、行数、中文编码正确；
- 导出中没有密码或绝对路径；
- 从备份恢复后记录数、关系和照片校验一致。

### 用户可见验收

下载并打开导出包；由 Codex 在独立空测试环境完成一次恢复，向用户报告恢复前后记录数和照片校验结果。

### 本阶段不做

- 网页中直接导入任意版本数据；
- 多币种汇率统计；
- 第三方云备份，除非用户明确指定并授权。

## 10. 阶段 7：PWA、登录加固与 Debian 13 部署

### 目标

把已稳定的本地应用安全部署到用户服务器，并提供接近 App 的手机入口。

### 交付物

- Manifest、图标、Service Worker 和离线提示；
- 手机与电脑响应式细节；
- 单用户登录、生产安全设置和限流；
- Gunicorn、Caddy、生产 Docker Compose；
- Debian 13 安装、升级、回退和日志说明；
- HTTPS、数据卷、健康检查和重启策略。

### 自动验证

- Django 生产安全检查；
- 未登录无法访问页面、照片和导出；
- HTTP 跳转 HTTPS，证书有效；
- 容器重启后数据库和照片仍存在；
- PWA 清单与 Service Worker 检查；
- 断网不允许产生虚假成功的写入。

### 用户可见验收

手机和电脑都能通过 HTTPS 登录；手机可添加到主屏幕；新增一条测试记录后重启容器，记录与照片仍存在。

### 本阶段不做

- 应用商店发布；
- 离线编辑和同步；
- 公开注册或社交分享。

## 11. 阶段 8：稳定化与 1.0 发布

### 目标

停止增加范围，集中验证真实日常使用和恢复能力。

### 交付物

- 全量回归测试；
- 手机和电脑核心旅程测试；
- 空状态、错误提示和危险操作检查；
- 第二次备份恢复演练；
- 文档收尾、已知限制和 1.0 Git 标签。

### 完成条件

- 全部自动测试通过；
- 没有已知数据丢失或高风险安全问题；
- 核心用户旅程全部通过；
- 用户亲自完成新增、再次品饮、筛选、照片查看、导出和登录；
- 备份能恢复；
- Git 工作区干净并建立 `v1.0.0` 标签。

### 本阶段不做

- 新功能；
- 未经风险评估的大规模重构；
- 为赶时间跳过恢复或安全检查。

## 12. 一个月建议安排

- 第 1 周：完成阶段 0、1 和 2，尽早得到能记录啤酒的版本。
- 第 2 周：完成阶段 3 和 4，加入评分、标签和照片。
- 第 3 周：完成阶段 5 和 6，加入筛选、统计和可恢复备份。
- 第 4 周：完成阶段 7 和 8，部署、手机实测和稳定化。

若进度落后，优先保证：Beer/Tasting 分离、核心记录、照片、登录、导出和完整备份。可推迟图表美化、动态评分管理界面、动态维度筛选和非关键视觉动画。

## 13. 用户当前需要做的操作

当前唯一阻塞是本机没有 Git 和 Docker。下一步由用户使用官方安装程序完成：

1. 安装 Git for Windows 64 位版；大多数安装选项可保留默认值。
2. 安装 Docker Desktop，按安装程序提示启用或更新 WSL 2，必要时重启电脑。
3. 重新打开 Codex 和 `D:\BEER-JORNAL` 项目。
4. 告诉 Codex：“工具已经安装，请重新检查环境，完成阶段 0 的 Git 初始化和文档提交；不要开始阶段 1。”

Codex 随后应验证 Git、Docker 与 Docker Compose，而不是让用户自行判断。Git 若要求用户名和邮箱，Codex必须向用户说明用途并由用户提供，不得编造个人信息。
