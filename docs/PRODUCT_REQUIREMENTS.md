# BEER JOURNAL 产品需求文档

文档状态：v2.0 重构设计已确认，待实现
更新日期：2026-07-12
目标版本：个人收藏 App v2.0

## 1. 产品定位

BEER JOURNAL 是一个仅供一位用户使用的私人啤酒资料库和品饮日志。用户可以在手机或电脑上记录喝过的啤酒、每次品饮的环境和感受，并通过筛选、排序和统计理解自己的偏好。

应用以中文为主，第一版是响应式 Web App / PWA，不发布到应用商店。正式版本部署在用户自己的 Debian 13 服务器，结构化数据和照片保存在该服务器并支持完整备份。

## 2. v2.0 已确认产品与界面决定

v2.0 将现有后台网页式体验重构为手机优先的私人啤酒收藏 App。以下决定优先于本文其余 1.0 基线描述；历史数据不因界面调整被删除或重解释。

| 主题 | v2.0 决定 |
|---|---|
| 视觉 | Apple 风格：极简、高级、大图片、大圆角、柔和阴影与短促自然动画。 |
| 信息架构 | 不再提供首页；启动后进入“我的啤酒”。底部 App 导航固定为“我的啤酒、饮用记录、个人数据”。 |
| 啤酒分类 | 使用两级分类：`BeerCategory`（拉格、艾尔；内部稳定代码仍为 `lager`、`ale`）→ `BeerStyle`（皮尔森、淡色拉格、黑拉格、IPA、小麦啤酒、世涛）。旧的泛称小类 “Lager” 不再提供；其已有啤酒安全迁移为“淡色拉格”。 |
| 风味标签 | 用户可在录入时直接输入自定义风味标签；规范化后去重并复用已有标签。 |
| 新增资料 | Beer 增加麦汁浓度 Plato 与口感档案（清爽/适中/醇厚）；Tasting 增加容量、饮用瓶数与购买渠道。 |
| 评分 | 日常 UI 与品饮业务流程只录入和展示总体评分；多维评分不再出现在新建、编辑、再次品饮或详情页。历史评分和数据结构继续保留。 |
| 标签 | 日常 UI 与品饮业务流程不再录入或展示食物搭配、饮用场景；历史品饮级标签继续保留，可在未来专门的历史/导出界面读取。 |
| 删除 | 不在主导航显示回收站；软删除与恢复数据能力继续保留，恢复入口改为低优先级设置/维护区域。 |
| 个人数据 | 新增“个人数据”页面，第一版展示啤酒款数、品饮次数、国家数、常喝类型、平均总体评分和月度品饮数量。 |

### v2.0 手机优先验收

- 三个底部导航页面在常见手机宽度下无横向滚动，主要触控区域不小于 44px。
- Beer 卡片优先展示图片、国旗、名称、类型和总体评分摘要；无图片显示安静占位。
- 表单改为少量字段的分步骤流程，并可在最后确认后一次提交。
- 页面进入、卡片点击与图片加载使用 150–250ms 的短动画；用户系统偏好“减少动态效果”时应禁用非必要动画。
- 国家以 ISO 国家代码映射为国旗及中文显示名称；无法映射时显示国家代码，不阻止记录。

### v3.0-A 设计系统

v3.0-A 只统一视觉基础，不改变 URL、数据库、Beer/Tasting 关系、表单字段或既有业务流程。设计采用暖灰页面背景、白色卡片、24px 卡片圆角、柔和阴影与移动端优先的触控尺寸；琥珀、绿色、蓝色、莓果粉和紫色仅用于评分、状态、分类和数据强调，避免大面积高饱和底色。所有非必要动效限制在 150–300ms，并遵从“减少动态效果”系统设置。

### v3.0-B1 收藏主页

“我的啤酒”在保留原有搜索、筛选、排序、详情链接和新增入口的前提下，升级为收藏主页：顶部显示未删除 Beer 数量、有效 Tasting 数量和有效总体评分平均值；卡片保持图片、国家、名称、品牌、两级分类、平均评分、风味标签和品饮次数。页面可显示“最近品饮”（最近 30 天有有效 Tasting）、“高评分”（平均总体评分不低于 8.5）和“新收藏”（最近 30 天创建）视觉标签。这些均为查询时计算的展示状态，不写入数据库。

## 2. 第一版成功标准

1. 用户能在手机上添加一款啤酒及第一次品饮记录。
2. 同一款啤酒能新增第二次、第三次品饮，历史数据互不覆盖。
3. 用户能编辑、查看、逻辑删除和恢复啤酒与品饮记录。
4. 一次品饮能上传多张照片，服务器只保留压缩后的安全版本及缩略图。
5. 用户能填写总体评分和品饮感想；历史多维评分仅保留用于兼容和导出。
6. 用户能使用多个条件组合筛选，并按评分、日期、名称或价格排序。
7. 统计页面能区分“啤酒款数”和“品饮次数”，且结果可用样本数据核对。
8. 用户能导出 JSON、CSV，并能将数据库与照片一起备份和恢复。
9. 手机可以把应用添加到主屏幕；没有网络时显示友好提示，不允许产生无法同步的离线修改。
10. 正式服务器必须通过 HTTPS 和单用户登录保护私人数据及照片。
11. 每个稳定阶段都有自动测试、实际操作验证、文档和可回退的 Git 提交。

## 3. 用户与使用场景

### 3.1 用户

- 只有应用所有者一位用户。
- 不提供公开注册、好友、社交关系或多用户权限。
- 用户没有编程经验，日常操作和错误提示必须使用清晰中文。

### 3.2 主要使用场景

- 在酒吧、餐厅或家中用手机快速记录当次品饮。
- 在电脑上补充啤酒资料、整理标签、查看照片和详细笔记。
- 找出某一国家、类型、价格或评分区间的历史记录。
- 查看月度记录数量、常喝类型、平均评分等个人统计。
- 将数据导出供人工查看，或完整备份到服务器之外的位置。

## 4. 已确定的产品决定

这些决定用于消除第一版歧义；如需改变，应先更新本文档再开发。

| 主题 | 第一版决定 |
|---|---|
| 记录模型 | `Beer` 保存一款啤酒的基本资料；`Tasting` 保存一次实际品饮 |
| 新建流程 | 新建一款“喝过的啤酒”时至少同时建立一次品饮；第一版不做愿望清单 |
| 主列表 | 同时提供“啤酒库”和“品饮日志”；高级筛选主要作用于品饮日志 |
| 登录 | 本地开发可简化，正式联网部署必须登录；关闭公开注册 |
| PWA | 支持安装、应用式显示、静态缓存和离线提示；所有数据操作要求联网 |
| 总体评分 | 用户独立填写，不自动由维度平均或加权覆盖 |
| 评分范围 | 总体及维度默认 0–10 分、0.5 分步进；未评分为留空，不是 0 分 |
| 评分维度 | 默认七项可停用、调整顺序并可在未来新增；旧记录保留当时的名称和量表 |
| 照片归属 | 照片属于一次品饮；啤酒详情汇总其历次品饮照片 |
| 标签归属 | 第一版标签属于啤酒；品饮场景标签以后再评估 |
| 酒精度 | 属于啤酒基本资料；不同酒精度的版本视为不同啤酒资料 |
| 容量与价格 | 属于每次品饮，容量统一保存为毫升，金额与货币代码分开保存 |
| 删除 | 普通删除进入回收站；永久删除必须二次确认并说明关联影响 |
| 完整导出 | JSON 保存可恢复的结构，CSV 供人工查看，照片随完整导出包提供 |

## 5. 核心数据概念

```text
Beer（啤酒基本资料）
  ├─ 1 对多 → Tasting（每次品饮）
  └─ 多对多 → Tag（标签）

Tasting
  ├─ 1 对多 → Photo（照片）
  └─ 1 对多 → TastingRating（多维评分值）

RatingDimension（评分维度）
  └─ 1 对多 → TastingRating
```

字段归属原则：

- 啤酒资料：名称、品牌、酒厂、国家或地区、产区、类型、酒精度、基础说明、标签。
- 品饮记录：饮用日期、饮用地点、容量、价格、货币、购买渠道、购买地点、总分、多维评分、品饮感想和照片。

## 6. 功能需求与验收标准

### FR-01 新增啤酒及首次品饮

用户可以在一个连贯流程中填写啤酒基本资料和第一次品饮记录。

验收标准：

- 啤酒名称和饮用日期为必填；日期默认当天并可修改。
- 可填写品牌、酒厂、国家或地区、产区、类型、酒精度、容量、价格、货币、购买渠道、购买地点、饮用地点和品饮感想。
- 数字范围、必填项和格式错误使用中文就地提示。
- 保存时啤酒资料和首次品饮应一起成功或一起失败，不能留下半条记录。
- 重复提交同一个表单不会静默生成两条完全相同的记录。
- 名称、品牌、酒厂相似时可以提示疑似重复，但不强制阻止合法的同名或不同批次记录。

### FR-02 再次品饮同一款啤酒

验收标准：

- 可从啤酒详情页新增另一条品饮记录。
- 新品饮拥有独立的日期、地点、容量、价格、购买信息、笔记、照片和评分。
- 新增或编辑一次品饮不会覆盖其他历史品饮。
- 啤酒详情按日期展示全部未删除品饮，并明确显示总次数。

### FR-03 浏览、详情、编辑、删除与恢复

验收标准：

- “啤酒库”以一款啤酒为一项；“品饮日志”以一次品饮为一项。
- 啤酒详情显示基本资料、标签、历次品饮和照片汇总。
- 编辑啤酒资料不会改写品饮历史。
- 普通删除将记录放入回收站，默认列表、筛选和统计不再显示。
- 可从回收站恢复；永久删除必须二次确认并说明会删除哪些评分、照片和关联数据。
- 有品饮历史的啤酒不能被无提示永久级联删除。

### FR-04 总体与多维评分

验收标准：

- 总体评分和每个维度评分都允许留空。
- 默认维度为香气、味道、口感、外观、回味、平衡度和性价比。
- 第一版评分范围是 0–10，步进为 0.5；超出范围或不符合步进时显示中文错误。
- 总体评分由用户单独填写；页面可以展示维度平均值，但不得自动覆盖用户总分。
- 停用、重命名或调整维度后，旧品饮仍按保存时的维度名称、范围和分值显示。
- 第一版可以只提供默认维度，不强制实现完整的维度管理页面，但数据库必须支持扩展。

### FR-05 标签

验收标准：

- 可以为啤酒新增、复用和移除自定义标签。
- 标签自动去除首尾空格，并以不区分大小写的方式避免重复。
- 删除标签关系不删除啤酒。
- 标签可以用于搜索和组合筛选。

### FR-06 多张照片与压缩

验收标准：

- 一次品饮可以上传一张或多张照片，支持预览、排序和删除。
- 支持的首批格式为 JPEG、PNG 和 WebP；HEIC/HEIF 仅在真实设备验证后加入。
- 上传端和服务器端都限制单张大小；服务器还限制解码后的像素数量。
- 服务器实际解码图片、修正方向、移除 EXIF、限制长边、重新编码，并生成缩略图。
- 默认不长期保留手机原图；压缩失败时不写入正式照片记录并清理临时文件。
- 未登录用户不能通过猜测地址读取照片。
- 永久删除照片后不遗留孤儿文件。

### FR-07 搜索

验收标准：

- 能按名称、品牌和酒厂进行不区分大小写的部分匹配。
- 能按国家或地区、类型和标签查找。
- 搜索输入为空时恢复正常列表，不报错。
- 没有结果时显示友好空状态和清除条件入口。

### FR-08 组合筛选

验收标准：

- 支持总体评分区间、酒精度区间、价格区间和饮用日期区间。
- 支持同时组合名称、品牌、酒厂、国家、类型、标签和上述区间。
- 不同类别之间默认“同时满足”；同一多选类别内部默认“满足任一项”。
- 价格区间只比较同一货币；第一版默认人民币 CNY。
- 未填写的值不参加对应区间筛选。
- 筛选条件写入 URL 查询参数，刷新或返回页面后不应无故丢失。
- 页面显示结果数量，并提供一键清空全部条件。

### FR-09 排序与分页

验收标准：

- 品饮日志支持按总体评分、饮用日期、啤酒名称和价格升序或降序。
- 空评分或空价格默认排在有值记录之后。
- 相同主排序值使用稳定的次排序，翻页时不能随机跳动。
- 列表必须分页，手机端翻页控件可触控且清楚显示当前页。

### FR-10 统计

验收标准：

- 同时显示啤酒款数和品饮次数，不能统称为一个“总记录数”。
- 国家或地区数量只统计非空、去重且至少有有效品饮的啤酒。
- 最常喝类型按品饮次数计算。
- 平均评分只计算填写了总体评分的有效品饮。
- 每月记录数量按实际饮用日期计算，不按创建时间计算。
- 所有统计排除回收站记录。
- 无数据时显示 0 或友好说明，不出现错误或误导性平均值。

### FR-11 JSON 与 CSV 导出

验收标准：

- JSON 导出包含版本号、导出时间、稳定 ID、所有结构化记录和关系，可用于迁移或恢复。
- CSV 按啤酒、品饮、评分、标签和照片清单分别导出，使用方便中文 Windows 表格软件打开的编码。
- 导出不包含密码、会话、数据库连接串、服务器绝对路径或其他秘密。
- 复杂关系的无损恢复以 JSON 为准，不宣称单一 CSV 能完整恢复全部关系。

### FR-12 完整备份与恢复

验收标准：

- 完整备份同时包含 PostgreSQL 备份、照片目录、迁移版本、清单和校验值。
- 至少保留一份服务器之外的副本；Docker 卷本身不视为备份。
- 提供可由 Codex 执行的备份和恢复步骤。
- 1.0 发布前必须在空测试环境完成一次真实恢复，并核对记录数和照片校验值。

### FR-13 PWA 与响应式体验

验收标准：

- 当前主流 Chrome、Edge、Safari 及手机浏览器中的核心流程可用。
- 可添加到手机主屏幕，拥有名称、图标、主题色和应用式显示。
- 主要按钮适合触控，表单在窄屏无需横向滚动。
- Service Worker 只缓存必要静态资源和离线提示，不缓存私人照片或离线提交写操作。
- 断网时明确提示需要联网，不让用户误以为数据已保存。

### FR-14 单用户登录

验收标准：

- 正式服务器没有公开注册入口，只保留一个由所有者控制的账户。
- 未登录用户无法访问列表、详情、照片、导出和统计。
- 登录后使用安全会话；登出后旧页面不能继续读取私人数据。
- 密码以安全散列形式保存，任何日志和导出都不包含密码。

## 7. 非功能需求

### NFR-01 简单与可维护

- 使用成熟、仍受支持且适合 Debian 13 和 Docker 的技术。
- 第一版不引入 Redis、消息队列、微服务或 Kubernetes。
- 单体项目保持一套数据模型、表单校验、登录和测试体系。

### NFR-02 数据完整性

- 使用数据库主外键、唯一约束、检查约束和事务保护重要关系。
- 所有结构变化有可追踪的数据库迁移。
- 失败的图片处理、导入或表单提交不能留下半条正式记录。

### NFR-03 安全与隐私

- 正式环境使用 HTTPS、安全 Cookie、CSRF 防护、主机名校验和登录失败限速。
- 秘密只通过环境变量或不提交的秘密文件提供。
- 默认不把任何私人内容传给第三方分析、AI、CDN 或存储服务。

### NFR-04 性能容量

- 以一位用户、约 10,000 款啤酒、50,000 次品饮及其照片作为宽松设计上限。
- 常用列表必须分页；个人常见数据量下普通页面目标为约 2 秒内可交互。
- 只有真实测量显示需要时才加入全文搜索扩展或缓存服务。

### NFR-05 易用与可访问

- 中文标签、校验和空状态清楚说明问题及解决方式。
- 表单控件有可见标签；状态不能只用颜色表达。
- 危险操作需要确认，重要默认值可以修改。

### NFR-06 可验证

- 每个阶段都有自动测试、数据库迁移检查和实际浏览器冒烟测试。
- 测试必须使用 PostgreSQL 环境。
- 不能用“代码看起来正确”替代运行验证。

### NFR-07 可运维

- 生产部署提供健康检查、清楚的错误日志、持久化数据卷和启动顺序。
- 备份、恢复、升级和回退均有文档及演练结果。

## 8. 第一版明确不做

- 社交、好友、评论、排行榜和公开分享。
- 多用户、开放注册和复杂权限。
- 原生 Android/iOS 应用及应用商店发布。
- AI 图片识别、自动品酒分析和条形码扫描。
- 地图、持续定位、路线或地点数据库。
- Untappd 或其他第三方啤酒数据库同步。
- 离线新增、离线编辑及跨设备冲突合并。
- 自动加权覆盖总体评分。
- 多币种汇率换算、库存和采购批次管理。
- Redis、消息队列、微服务、Kubernetes 或独立搜索服务。

## 9. 主要风险与处理

| 风险 | 处理方式 |
|---|---|
| 把啤酒资料与品饮混在一起 | 从第一条迁移起严格分为 Beer 和 Tasting，并在界面提供两个视图 |
| 中途频繁加功能导致范围失控 | 每次只推进一个阶段，新增需求先分析影响并更新文档 |
| 调整评分维度破坏历史 | 维度只能停用；评分值保存名称和量表快照 |
| 照片迅速占满服务器 | 限尺寸、重新编码、缩略图、默认不保留原图、定期孤儿检查 |
| 只备份数据库导致照片丢失 | 数据库与照片必须成套备份，并保留异机副本 |
| 私人照片绕过登录 | 照片经鉴权视图读取，不使用公开媒体目录 |
| 同名啤酒重复或误合并 | 第一版只提示疑似重复，不设置可能误伤的强唯一约束 |
| 不会编程，难以判断是否完成 | 每阶段给出可见验收步骤、预期结果和实际测试报告 |

## 10. 1.0 发布判定

只有以下条件全部满足，才将版本标记为 1.0：

- FR-01 至 FR-14 的验收标准完成，或有用户明确接受的书面例外；
- 自动测试全部通过，核心流程已在真实手机和电脑上操作验证；
- 正式环境安全检查无未解释的高风险项；
- Debian 13 服务器重启容器后数据仍存在；
- 从完整备份恢复到空环境的演练成功；
- 用户亲自完成一次新增、再次品饮、筛选、照片查看、导出和登录验证；
- 文档与实际实现一致，并建立可回退的 Git 版本标签。
